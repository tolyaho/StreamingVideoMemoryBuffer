## 8. Actually Long Video

The whole point of a hierarchical memory is to keep answering questions correctly after content has rolled out of the recent window, and 17 min (the video I tested on) doesn't really stress that. So for this section I grabbed one of the LVBench videos — YouTube id `16Z-XQh9jhk`, an 84 min live concert — which is handy because LVBench ships QAs with it, $22$ of them for this clip, covering entity recognition, event understanding, temporal grounding and reasoning. I pulled them out of `video_info.meta.jsonl` and reformatted into the same `question / options / answer / time_stamp` shape I've been using, saved to `data/lvbench/16Z-XQh9jhk/qas.jsonl`.

This time I also switch on the persistence layer I wrote in `src/memory_db.py` (`MemoryStore`, peewee/SQLite) — windows, episodes and events get flushed to `outputs/memory_16Z-XQh9jhk.db` as they're built. It's a durable sidecar, not a RAM offload: retrieval still runs against the in-memory writer state, and the window archive stays in Python for the whole run. The db is for post-run inspection and for being able to pick up where I left off if a long run dies halfway through.

### Results

End-to-end I get $8/22$ ($36.4\%$) on this clip, which I'm actually pretty happy with for a setup that never shows pixels to the reasoner. A lot of the misses are on questions that are just genuinely brutal for a text-only LLM reading short captions — "what colour are her earrings", "what colour is her floor mic stand at some specific minute", "how many white lights are on at $45{:}08$", "what colour did she change the microphone to". The captioner doesn't reliably mention jewellery or count small light sources, so the evidence the LLM gets simply doesn't contain the answer; retrieval can't recover information that was never written down. On the questions that are closer to "what happened / who appeared / what were the dancers doing" the system does fine, which is the regime the hierarchy is actually designed for.

Worth saying out loud that this is a demo, not a benchmark number — one video, $22$ QAs, a small text-only reasoner ($3$B), and a caption-only evidence channel. Closing the gap on the fine-grained visual questions is mostly a captioner / visual-context problem, not a memory problem: swap in a stronger frame VLM (or route `visual_context` into the reasoner) and most of those earring/colour/count questions come back. The memory structure itself is holding up fine over $84$ min — $1597$ windows promoted, $1071$ episodes flushed, $519$ long-term events consolidated, and latency stays flat throughout the run.
